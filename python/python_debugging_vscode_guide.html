1. Introduction to Debugging

Debugging is the process of identifying and removing errors from your code. It's a critical skill for every Python developer. While print statements can be useful for simple issues, a proper debugger provides much more power and insight.

When to Use a Debugger vs. Print Statements

Use Print Statements When:
  • Quick verification of values in a known location
  • Logging application behavior in production
  • Debugging simple, linear code flow
  • Creating permanent logging for monitoring
Use the Debugger When:
  • Investigating complex logic or control flow
  • Examining object state and relationships
  • Stepping through loops and conditionals
  • Analyzing exception stack traces
  • Understanding third-party library behavior
  • Debugging multi-threaded or async code

Prerequisites

  1. Install Python Extension
    • Open VS Code

3. Debugging Basics

Starting a Debug Session

There are multiple ways to start debugging in VS Code:

  1. Method 1: Run and Debug View
    • Click the Run and Debug icon in the Activity Bar (left sidebar)
    • Click "Run and Debug" button
    • Select "Python File" from the dropdown
  2. Method 2: Keyboard Shortcut
    • Press F5 to start debugging
    • First time: Select "Python File" debug configuration
  3. Method 3: Right-click Context Menu
    • Right-click in the editor
    • Select "Run Python File in Debugger"

The Debug Interface

When debugging starts, VS Code shows several panels:

Panel Location Purpose
Variables Left sidebar Shows local and global variables with their current values
Watch Left sidebar Monitor specific expressions or variables
Call Stack Left sidebar Shows the execution path (function calls)

Debug Toolbar Top center Control execution flow (step, continue, stop)

4. Working with Breakpoints

Setting Breakpoints

Breakpoints pause execution at specific lines of code.

How to Set Breakpoints:
  1. Click in the gutter (left of line numbers) - red dot appears
  2. Press F9 with cursor on desired line
  3. Right-click line number ‚Üí "Add Breakpoint"

Types of Breakpoints

Regular Breakpoints

Pause execution every time the line is reached.

def calculate_total(items): total = 0 # ‚Üê Set breakpoint here for item in items: total += item['price'] return total

Conditional Breakpoints

Only pause when a condition is true.

  1. Right-click on a breakpoint (red dot)
  2. Select "Edit Breakpoint..."
  3. Choose "Expression" or "Hit Count"
# Example: Break only when price > 100 def process_items(items): for item in items: price = item['price'] # Conditional: price > 100 process(price)
üí° Conditional Breakpoint Examples:
  • i == 50 - Break on 50th iteration
  • len(data) > 1000 - Break when list exceeds 1000 items
  • user.id == '12345' - Break for specific user
  • error_count > 5 - Break after multiple errors

Logpoints

Log messages to console without pausing execution.

  1. Right-click in gutter
  2. Select "Add Logpoint..."
  3. Enter message with expressions in curly braces
# Logpoint message: "Processing user {user.name} with balance {balance}" def process_user(user): balance = get_balance(user) # Logpoint here logs without stopping update_account(user, balance)

Exception Breakpoints

Pause when exceptions are raised.

  1. Open Breakpoints panel
  2. Check "Raised Exceptions" or "Uncaught Exceptions"
  3. Debugger pauses when exception occurs

6. Inspecting Variables and State

Variables Panel

Shows all variables in current scope organized by category:

  • Locals: Variables in current function
  • Globals: Module-level variables
  • Special Variables: Python internals (like __name__, __file__)
üí° Variables Panel Features:
  • Expand objects: Click arrow to see attributes and methods
  • View collections: Lists, dicts, sets show all elements
  • Modify values: Double-click value to change during debugging
  • Copy value: Right-click ‚Üí Copy Value

Watch Expressions

Monitor specific expressions or variables across execution.

  1. Click "+" in Watch panel
  2. Enter expression to monitor
  3. Value updates as you step through code
# Useful watch expressions: len(items) # Monitor collection size user.is_active # Track boolean state sum(order.values()) # Calculate totals type(variable) # Check type changes id(object) # Track object identity

Hover Evaluation

Hover over any variable in the editor to see its current value and type.

Advanced Hover Tips:
  • Hover over expressions like user.name to evaluate them
  • Hover shows rich formatting for complex objects
  • Click small arrow in hover popup to add to Watch

7. Debug Console

Using the Debug Console

The Debug Console is a Python REPL that runs in your paused program's context.

What You Can Do in Debug Console:
  • Evaluate expressions: price * 1.1
  • Call functions: calculate_total(items)
  • Inspect objects: dir(user), vars(obj)
  • Test fixes: new_value = old_value * 2
  • Import modules: import json; json.dumps(data)
  • Modify state: user.balance = 1000

Debug Console Examples

# When paused at a breakpoint, try these in Debug Console: # Inspect variable types type(items) # Output: # Check object attributes dir(user) # Output: ['__class__', ..., 'email', 'name', 'id'] # Test transformations [item['price'] * 1.1 for item in items if item['price'] > 10] # See transformed data without modifying code # Call helper functions import json json.dumps(data, indent=2) # Pretty print data structures # Temporarily fix bugs to test user.balance += 100 # Continue execution with modified state
‚ö†Ô∏è Warning: Changes made in Debug Console only affect the current debug session. They don't modify your source code.

9. Advanced Debugging Techniques

Post-Mortem Debugging

Debug after a crash using the Python debugger.

import sys import pdb def problematic_function(): data = [1, 2, 3] return data[5] # Will raise IndexError if __name__ == "__main__": try: problematic_function() except Exception: # Enter debugger at point of exception pdb.post_mortem()

Remote Debugging

Debug Python code running on another machine or container.

# 1. Install debugpy in your remote environment pip install debugpy # 2. Add to your Python code: import debugpy # Allow other machines to attach debugpy.listen(('0.0.0.0', 5678)) print("Waiting for debugger attach...") debugpy.wait_for_client() debugpy.breakpoint() # Your code here print("Debugger attached!")
// 3. Create attach configuration in VS Code: { "name": "Python: Remote Attach", "type": "python", "request": "attach", "connect": { "host": "remote-server-ip", "port": 5678 }, "pathMappings": [ { "localRoot": "${workspaceFolder}", "remoteRoot": "/app" } ] }

Multi-threaded Debugging

Debug applications with multiple threads.

import threading import time def worker(name): for i in range(5): # Set breakpoint here to debug specific thread print(f"{name}: {i}") time.sleep(1) # Create threads t1 = threading.Thread(target=worker, args=("Thread-1",)) t2 = threading.Thread(target=worker, args=("Thread-2",)) t1.start() t2.start()
üí° Thread Debugging Tips:
  • Call Stack panel shows all threads
  • Click thread to switch context
  • Set conditional breakpoints: threading.current_thread().name == "Thread-1"
  • Use "Pause on All Threads" in debug toolbar dropdown

Debugging Generators and Iterators

def fibonacci_generator(n): a, b = 0, 1 for _ in range(n): # Breakpoint here to inspect generator state yield a a, b = b, a + b # Debugging generator execution gen = fibonacci_generator(10) for num in gen: # Step through each iteration print(num)

Debugging Decorators

import functools import time def timing_decorator(func): @functools.wraps(func) def wrapper(*args, **kwargs): start = time.time() # Breakpoint here to inspect decorator behavior result = func(*args, **kwargs) end = time.time() print(f"{func.__name__} took {end - start:.2f}s") return result return wrapper @timing_decorator def slow_function(): time.sleep(1) return "Done" # Stepping into decorated function shows decorator flow slow_function()

Debugging Async Code

import asyncio async def fetch_data(url): # Breakpoint in async function await asyncio.sleep(1) return f"Data from {url}" async def main(): tasks = [ fetch_data("url1"), fetch_data("url2"), fetch_data("url3") ] # Step through async execution results = await asyncio.gather(*tasks) return results # Run async code asyncio.run(main())
üí° Async Debugging Tips:
  • Set breakpoints in async functions normally
  • Call stack shows async execution flow
  • Watch expressions work with awaitable objects
  • Debug console can use await keyword

Debugging Context Managers

class DatabaseConnection: def __enter__(self): # Breakpoint to debug resource acquisition print("Opening connection") self.conn = "connection_object" return self.conn def __exit__(self, exc_type, exc_val, exc_tb): # Breakpoint to debug cleanup print("Closing connection") if exc_type: print(f"Exception: {exc_val}") return False with DatabaseConnection() as conn: # Breakpoint here to inspect connection usage print(f"Using {conn}")

10. Common Debugging Scenarios

Scenario 1: Debugging Infinite Loops

def process_queue(queue): while queue: # Set breakpoint here item = queue.pop(0) process(item) # Use conditional breakpoint: len(queue) > 1000 # or watch expression: len(queue)
Strategy:
  1. Set conditional breakpoint to trigger after many iterations
  2. Add watch expression to monitor loop variable
  3. Check loop exit condition in Debug Console
  4. Use "Pause" button (‚è∏Ô∏è) to break into running loop

Scenario 2: Finding Where Variable Changes

def complex_processing(data): result = [] # Want to know where 'result' gets modified? # 1. Add 'result' to Watch expressions # 2. Step through and observe when value changes # 3. Use Debug Console: id(result) to track object identity for item in data: transform1(item, result) transform2(item, result) transform3(item, result) return result
Strategy:
  1. Add variable to Watch panel
  2. Set breakpoints at suspected modification points
  3. Use logpoints to log variable state: "result = {result}"
  4. Check id() in Debug Console to detect reassignment

Scenario 3: Debugging Function Called Many Times

def calculate_price(item, discount): # Called 10,000 times, only interested when discount > 0.5 price = item['base_price'] discounted = price * (1 - discount) return discounted # Set conditional breakpoint: discount > 0.5 for item in items: price = calculate_price(item, get_discount(item))
Strategy:
  1. Use conditional breakpoint with specific condition
  2. Or use hit count: Break on 500th hit
  3. Or use logpoint to collect data without stopping

Scenario 4: Debugging Exceptions

def process_data(data): try: result = transform(data) validate(result) return save(result) except Exception as e: # Set breakpoint here to inspect exception logger.error(f"Error: {e}") raise
Strategy:
  1. Enable "Raised Exceptions" in Breakpoints panel
  2. Debugger pauses when exception is raised
  3. Inspect call stack to see execution path
  4. Check variables to understand why exception occurred
  5. Use Debug Console to test fixes

Scenario 5: Debugging Third-Party Library Code

import requests def fetch_api_data(url): response = requests.get(url) # Want to debug requests library? # Use Step Into (F11) to enter requests code return response.json()
Strategy:
  1. Use Step Into (F11) to enter library code
  2. Set breakpoints in library files (find them via Step Into)
  3. Enable "justMyCode": false in launch.json to debug library code
  4. Use call stack to navigate through library execution
// launch.json configuration for debugging libraries: { "name": "Python: Debug Libraries", "type": "python", "request": "launch", "program": "${file}", "justMyCode": false, // Enable debugging of library code "console": "integratedTerminal" }

Scenario 6: Debugging List Comprehensions

# Complex list comprehension - hard to debug result = [transform(item) for item in data if validate(item)] # Better for debugging - use regular loop with breakpoints result = [] for item in data: if validate(item): # Breakpoint here transformed = transform(item) # Breakpoint here result.append(transformed)
üí° Tip: Convert complex comprehensions to regular loops when debugging, then optimize back to comprehensions when fixed.

Scenario 7: Debugging Data Transformations

def process_pipeline(data): # Track data transformation through pipeline step1 = [x * 2 for x in data] # Breakpoint + watch: step1 step2 = [x + 10 for x in step1] # Breakpoint + watch: step2 step3 = [x for x in step2 if x > 20] # Breakpoint + watch: step3 return step3 # Debug Console commands at each breakpoint: # len(step1), min(step1), max(step1) # [x for x in step2 if x < 0] # Find unexpected values

Scenario 8: Debugging Class Instances

class Order: def __init__(self, items): self.items = items self.total = 0 def calculate_total(self): # Breakpoint here self.total = sum(item['price'] for item in self.items) return self.total # When paused: # - Hover over 'self' to see all attributes # - Expand 'self' in Variables panel # - Debug Console: vars(self), dir(self) # - Watch: self.total, len(self.items)

11. Best Practices and Tips

Debugging Workflow Best Practices

1. Start with the Error Message
  • Read the full traceback carefully
  • Identify the exact line where error occurred
  • Understand the error type and message
  • Set breakpoint at or before the error line
2. Use the Scientific Method
  1. Observe: What is the actual behavior?
  2. Hypothesize: What might be causing it?
  3. Test: Set breakpoints to verify hypothesis
  4. Iterate: Refine hypothesis based on findings
3. Narrow Down the Problem
  • Binary search: Set breakpoint in middle of suspected code
  • If bug is before, move breakpoint earlier
  • If bug is after, move breakpoint later
  • Repeat until you find exact location
4. Inspect Data, Not Just Code
  • Check variable types: type(variable)
  • Verify data structure: len(), .keys(), .items()
  • Look for None, empty strings, zero values
  • Check for type mismatches (string vs int)
5. Use Logpoints for High-Frequency Code
  • Functions called many times
  • Loop iterations
  • Event handlers
  • Async callbacks

Performance Debugging Tips

# Use time module to identify slow sections import time def slow_function(): start = time.time() # Breakpoint + watch: start # Expensive operation 1 result1 = expensive_op1() time1 = time.time() - start # Watch: time1 # Expensive operation 2 result2 = expensive_op2() time2 = time.time() - start - time1 # Watch: time2 return result1 + result2
üí° Performance Debugging Strategy:
  1. Add timing watches between operations
  2. Use cProfile for detailed profiling
  3. Set breakpoints before/after slow sections
  4. Inspect data sizes: len(data), sys.getsizeof(obj)

Memory Debugging Tips

import sys def check_memory_usage(): data = load_large_dataset() # Breakpoint here - check memory in Debug Console: # sys.getsizeof(data) # len(data) # import gc; gc.get_count() process(data)

Common Debugging Mistakes to Avoid

‚ùå Don't:
  • Set too many breakpoints: Makes debugging slow and confusing
  • Ignore the call stack: Shows valuable execution context
  • Forget to check variable types: Type mismatches are common bugs
  • Debug with production data: Use small, controlled test data
  • Modify code while debugging: Can cause inconsistent behavior
  • Skip exception details: Exception messages contain crucial info
‚úÖ Do:
  • Use minimal test cases: Smallest data that reproduces bug
  • Document your findings: Take notes on what you discover
  • Test your fixes: Run through debugger to verify
  • Learn from bugs: Understand why the bug occurred
  • Clean up breakpoints: Remove them when done

Debugging Complex Conditions

12. Troubleshooting

Debugger Won't Start

Problem: Pressing F5 does nothing or shows error

Solutions:
  1. Check Python extension is installed
    • Extensions ‚Üí Search "Python" ‚Üí Install
  2. Select correct Python interpreter
    • Ctrl+Shift+P ‚Üí "Python: Select Interpreter"
  3. Verify Python is in PATH
    • Terminal: python --version
  4. Check launch.json syntax
    • No trailing commas, valid JSON
  5. Restart VS Code
    • Sometimes extension needs reload

Breakpoints Not Hitting

Problem: Breakpoint is set but not triggered

Solutions:
  1. Verify breakpoint is enabled
    • Should be solid red dot, not gray
  2. Check if code is executed
    • Add print statement to verify
  3. Verify correct file is running
    • Check terminal output for file path
  4. Check justMyCode setting
    • Set "justMyCode": false for library code
  5. Restart debugger
    • Stop and start again (Shift+F5, then F5)

Variables Not Showing Values

Problem: Variables panel empty or showing wrong values

Solutions:
  1. Check you're paused at breakpoint
    • Status bar should show "Paused on breakpoint"
  2. Expand variable categories
    • Click arrows next to Locals, Globals
  3. Check variable scope
    • Variable must be in scope at breakpoint
  4. Use Debug Console
    • Manually evaluate: print(variable)

Debugger Too Slow

Problem: Debugging is very slow

Solutions:
  1. Reduce number of breakpoints
    • Keep only essential ones
  2. Use conditional breakpoints
    • Instead of breaking every iteration
  3. Enable justMyCode
    • Set "justMyCode": true in launch.json
  4. Use smaller test data
    • Reproduce bug with minimal dataset
  5. Disable watch expressions
    • Remove expensive evaluations

Can't Debug in Virtual Environment

Problem: Debugger can't find modules in virtualenv

Solutions:
  1. Select virtualenv interpreter
    • Ctrl+Shift+P ‚Üí "Python: Select Interpreter" ‚Üí Choose venv
  2. Activate venv in launch.json
    • Use correct Python path in configuration
  3. Check venv is activated in terminal
    • Should see (venv) in prompt
// launch.json for virtual environment { "name": "Python: Virtual Env", "type": "python", "request": "launch", "program": "${file}", "python": "${workspaceFolder}/venv/bin/python", // Explicit path "console": "integratedTerminal" }
